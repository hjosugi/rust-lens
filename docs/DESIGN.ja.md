# Rust所有権レンズ設計

Rust所有権レンズは1つのルールに基づいて構築されています: **コンパイラが真実の源であり、拡張機能が翻訳者である**。拡張機能は借用チェックを再実装しようとするべきではありません。Rust自身のツールを実行し、構造化された診断を読み取り、それらの診断を学習者に優しい説明、タイムライン、焦点を絞ったエディタアクションに変換します。

## 目標

- 実際の `rustc` の所有権、借用、ライフタイム、イテレータ、および非同期診断を平易な言葉で説明する。
- 説明を `cargo check --message-format=json` に基づいてコンパイラの権威を保持する。
- エラーが発生した理由をrustcスパンに結びつけたコンパクトなASCIIタイムラインを通じて示す。
- rust-analyzer、Cargo、Clippy、またはrustcドキュメントを置き換えることなく、実際のプロジェクト内で有用であり続ける。

## アーキテクチャフロー

```text
cargo check --message-format=json
  -> JSON行パーサー
  -> 診断抽出/フィルタ
  -> 所有権/ライフタイムテンプレート
  -> スパンイベントタイムライン
  -> Webviewパネル / OutputChannel / 問題統合
```

同じパーサー/レポートパスは `rust-lens explain` からも利用可能であり、エディタに依存しない消費者はVS Codeを起動せずにcargo/rustc JSONをツールにパイプできます。

現在の実装の詳細:

- `runCargoCheck` はアクティブなワークスペースで構成されたcargoコマンドを実行します。
- `parseCargoJsonMessages` はCargoでラップされたコンパイラメッセージと直接のrustc JSON診断を受け入れ、`$message_type`の有無にかかわらず診断を含みます。
- `extractDiagnostics` はデフォルトでエラーをフィルタリングし、設定を通じて警告を含めることができます。
- `templateForDiagnostic` は既知のエラーコードを説明と修正ガイダンスにマッピングします。
- `buildEvents` と `renderEventTimeline` はrustcスパンと子スパンを順序付けられたタイムライン行に変換します。
- `RustLensViewProvider` はVS Code側のWebviewにレポートをレンダリングします。
- `Rust Ownership Lens` OutputChannel は生のcargo出力と生成されたレポートを保存します。
- 問題統合は将来の `DiagnosticCollection` サポートのための自然なプレゼンテーション境界です; すべての問題エントリはrustc診断から派生するべきであり、独立した借用チェッカーの推測からではありません。

## モジュール境界

プロトタイプは現在 `extension.js` に存在しますが、論理的な境界は次のとおりです:

- **Cargoランナー:** ワークスペース検出、信頼されたワークスペース設定の処理、コマンドの生成、JSONメッセージ形式のチェック。
- **パーサー:** 行指向のJSONパースとCargoおよびrustc診断形状との互換性。
- **診断モデル:** コード抽出、主要スパン選択、パス正規化、ソーススニペット、変数推測。
- **説明者:** エラーコードテンプレート、イテレータ移動の特化、rustc子メッセージ、フォールバックコンパクト診断。
- **タイムライン:** スパン分類（`borrow`、`move`、`drop`、`use`、`return`、`primary error`）とASCIIレンダリング。
- **プレゼンテーション:** Webview HTML、OutputChannelテキスト、ステータスバー状態、ホバーヒント、コードアクション、将来の問題ブリッジ。
- **CLI:** `bin/rust-lens.js` は `rust-lens explain [--json]` のためのパーサー/レポートビルダーの薄いエントリポイントです。
- **ヒューリスティックス:** 選択されたコードの説明と軽量ホバー。これらは明確にヒューリスティックとしてラベル付けされ、cargo出力に対して二次的である必要があります。
- **テストと互換性チェック:** パーサー/設定/テンプレートの動作に対するNodeテストと実際のrustc JSONに対するスクリプトチェック。

将来のリファクタリングは、新機能を追加する前にこれらの境界を保持する必要があります。

## サポートされているエラーコード

| コード | 現在の説明の焦点 |
| --- | --- |
| `E0106` | ライフタイム指定子が欠落; 返されたり保存されたりする参照がどこから来るべきかを説明します。 |
| `E0382` | 移動後の使用または借用; 専用の `for item in collection` イテレータ移動パスを含みます。 |
| `E0499` | 複数の重複する可変借用。 |
| `E0502` | アクティブな不変借用と衝突する可変借用。 |
| `E0505` | 値がまだ借用されている間の移動。 |
| `E0515` | ローカルデータへの参照を返す。 |
| `E0597` | 借用された値が十分に長く生存しない。 |
| `E0716` | まだ借用されている間に一時的な値がドロップされる。 |

未知の診断はコンパクトなコンパイラ診断として表示され、ユーザーが情報を失わないようにします。

## ロードマップ

### 基盤

- rustc/Cargo JSONを真実の境界として保持する。
- パーサーの互換性をrustc JSON形状とワークスペースツールチェーンのオーバーライドに広く保つ。
- 機能の成長が必要な場合、上記のモジュール境界に沿って単一ファイルのプロトタイプを分割する。

### MVP

- サポートされている所有権/ライフタイム診断のためにWebviewとOutputChannelレポートを維持する。
- 選択されたコードとホバーの説明を明確にヒューリスティックとしてマークする。
- rustcがすでに提案を提供している場合にのみ、高信頼のクイックフィックスを追加する。

### タイムライン、非同期、トレイト

- マルチファイル診断とマクロ展開されたスパンのためにスパンタイムラインを改善する。
- `Send`、`'static`、`async move`、および生成されたタスクのライフタイムのための非同期特有のテンプレートを追加する。
- rustcが十分な構造化されたコンテキストを提供する場合にトレイト/オブジェクトライフタイムの説明を追加する。

### Cargo、マクロ

- 実際のコマンドを隠すことなく `cargo check`、`cargo clippy`、およびターゲットパッケージ/bin/test引数をサポートする。
- マクロが多い診断を処理し、関連するスパンをユーザーが作成した呼び出しサイトに戻るようにします。
- キャンセル、デバウンス、およびワークスペース信頼動作が明示的になるまでバックグラウンドウォッチャーを実行しない。

### CLI、LSP

- CLI契約を安定させ、その後、公開形状が安定したときにパーサーとレポートビルダーを専用パッケージに抽出する。
- コアフォーマットが安定した後にのみLSPまたは診断ブリッジを検討する。
- Webviewで使用されるのと同じコンパイラ由来のモデルから問題エントリと機械可読出力を公開する。
